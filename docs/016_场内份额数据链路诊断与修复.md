## 三、根因分析（3 个 Bug）

### Bug 1：VPS 抓取范围不足（主因）

**现象**：88 只基金中只有 50 只有份额数据，34 只完全缺失

**根因**：VPS 上的 `jsl_vps_symbols.txt` 只包含 50 只深交所基金 + 1 只上交所基金（sh501018）。上交所的 511880、511360、511520、501025 等全部不在抓取列表中。

**证据**：
```
VPS jsl_vps_symbols.txt 旧内容（2行）:
161716,168204,...,165513   (49只深交所)
,sh501018                   (1只上交所)
```

VPS 脚本 `041_jsl_cloud_shares.py` 实际上**已经内置了上交所批量获取能力**（`fetch_sse_shares_all()` 函数，一次获取 119 只 LOF 份额），但 symbols 列表中没有包含上交所基金代码，导致这些基金从未被采集。

**修复**：将 `jsl_vps_symbols.txt` 扩展为 81 只基金（55 深交所 + 26 上交所），已上传 VPS。

### Bug 2：本地 step9 同步逻辑过于保守

**现象**：即使 VPS 有历史数据，本地也经常不拉取

**根因**：step9 有两个过于激进的 guard：

```python
# Guard 1: 今日已标记就跳过（一旦标记永不回头）
if self.db.is_access_synced_today(today_str, source='jsl_shares_data'):
    return

# Guard 2: 昨日数据已在库就跳过（但今天的数据没入库）
if count > 0:  # 昨日份额已在库
    return
```

Guard 2 尤其有害：如果 06-16 的份额已在库，step9 就认为"份额已同步"直接跳过，但 06-17 的份额根本没拉取。

**修复**：重写 step9，去掉两个 guard，改为逐文件检查：对每个 VPS 文件，检查 DB 中该日期是否已有 >= 30 条份额数据，没有才入库。

### Bug 3：日期对齐问题（待确认）

**现象**：VPS 文件名是采集日期（如 `shares_2026-06-17.json`），但 06:00 采集时交易所尚未开盘

**分析**：
- VPS 06:00 采集 → 交易所 API 返回的是 T-1 日收盘后的份额（因为今天还没开盘）
- step9 按 `file_date`（= 采集日期）写入 DB → 实际写入了 T-1 的数据到 T 的日期

**实际影响**：目前影响不大，因为份额数据变化缓慢（日变化 < 1%）。但严格来说日期应该对齐到 T-1。

**修复状态**：暂未修改日期逻辑，因为需要确认交易所 API 在 06:00 返回的到底是哪天的数据。如果确认是 T-1，需要在 step9 中将 file_date 减一天。

## 五、VPS 脚本架构

### `041_jsl_cloud_shares.py`（VPS 端）

**部署位置**：`/root/LOFarb/041_jsl_cloud_shares.py`
**Cron 触发**：`0 6 * * 1-5`（工作日 06:00）
**输入**：`/root/LOFarb/jsl_vps_symbols.txt`（逗号分隔，SH 基金加 `sh` 前缀）
**输出**：`/root/LOFarb/siphon_data/shares_YYYY-MM-DD.json`

**采集逻辑**：
1. 深交所基金（16xxxx）：逐只调用 `szse.cn/api/report/ShowReport/data` 接口，每只间隔 3-6 秒（反爬）
2. 上交所基金（5xxxxx + sh 前缀）：调用 `query.sse.com.cn/commonQuery.do` 批量获取 119 只 LOF 份额，一次搞定
3. 两个交易所的数据合并为一个 JSON 文件

**关键字段**：
- 深交所：`dqgm`（当前规模，单位万份）
- 上交所：`INTERNAL_VOL`（内部规模，单位万份）

### `daily_updater.py step9`（本地端）

**调用链**：
```
step9_fetch_jsl_shares_from_vps()
  → _try_sync_all_from_vps('shares')
    → SFTP 列出 VPS /root/LOFarb/siphon_data/shares_*.json
    → 下载到本地 data/vps_sync/ 目录
    → 返回 [{date, content}, ...]
  → 逐文件检查 DB 中该日期份额数量
  → 缺失的写入 unified_fund_history.shares
```
